# 유틸리티, 헬퍼, 내부 메서드

## 1. 핵심 차이 요약 (한눈에 보기)

가장 쉬운 구분법은 **"이 메서드를 누가 호출하는가?"**를 보는 것입니다.

| 구분 | 별칭 | 핵심 키워드 | 누가 쓰는가? | 상태(State) 유무 |
| :--- | :--- | :--- | :--- | :--- |
| **유틸리티(Utility)** | 공구함 | **범용성, 정적(Static)** | **누구나** (프로젝트 전체) | 없음 (Stateless) |
| **헬퍼(Helper)** | 조수 | **보조, 가독성** | **특정 클래스/기능** | 있을 수도 있음 |
| **내부(Private)** | 부속품 | **캡슐화, 숨김** | **나 자신** (해당 클래스만) | 상관없음 |

---

### 2. 상세 설명 및 예시

#### ① 유틸리티 메서드 (Utility Method)
* **성격:** 프로젝트 **어디서든** 가져다 쓸 수 있는 "공공재"입니다. 특정 비즈니스 로직과 무관한 순수 기능을 담당합니다.
* **특징:**
    * 주로 `static`으로 선언하여 인스턴스 생성 없이 씁니다.
    * 데이터를 저장하지 않습니다(Stateless). 입력(`input`)이 같으면 항상 같은 출력(`output`)이 나옵니다.
* **예시:** 날짜 포맷 변환, 문자열 자르기, 수학 계산 등.
    > "누가 쓰든 `Math.max(10, 20)`은 항상 20이어야 해."



#### ② 헬퍼 메서드 (Helper Method)
* **성격:** 어떤 **"주요 작업"을 도와주는 조수** 역할입니다. 메인 로직이 너무 길어지거나 복잡해질 때, 가독성을 위해 로직의 일부를 떼어낸 것입니다.
* **특징:**
    * 범용적이지 않고, **특정 문맥(Context)** 안에서만 의미가 있습니다.
    * 다른 곳에서 재사용하기 위해 만든다기보다, **원래 메서드를 깔끔하게 만들기 위해** 만듭니다.
* **예시:** `로그인()`이라는 큰 메서드 안에서 `비밀번호_유효성_검사()`를 따로 떼어내는 경우.
    > "`비밀번호_유효성_검사`는 `로그인` 로직을 돕기 위해 존재해. 쇼핑몰 결제 로직에서 이걸 쓸 일은 거의 없어."

#### ③ 내부 메서드 (Internal/Private Method)
* **성격:** **"접근 제어(Visibility)"**의 관점입니다. 클래스 밖에서는 절대 부를 수 없고, 클래스 내부에서만 쓰이는 메서드입니다.
* **특징:**
    * 외부에 공개하고 싶지 않은 **구현 상세(Detail)**를 숨길 때 씁니다.
    * 사실상 **대부분의 헬퍼 메서드는 내부(Private) 메서드로 구현**됩니다.
* **예시:** 자동차의 `가속()`은 운전자가 밟는 페달(Public)이지만, 그 안에서 작동하는 `연료_분사()`는 엔진 내부(Private)에서만 일어납니다.

### 3. 관계 정리 (벤 다이어그램처럼 생각하기)

사람들이 헷갈리는 이유는 **교집합** 때문입니다.

1.  **헬퍼 메서드이자 내부 메서드인 경우 (가장 흔함):**
    * 특정 클래스의 복잡한 로직을 돕기 위해 만들었고(`Helper`), 굳이 밖에서 알 필요가 없어서 `private`으로 막아둔 경우(`Internal`).

2.  **퍼블릭 헬퍼 메서드인 경우:**
    * 다른 클래스에서도 이 "도우미" 기능이 필요해서 `public`으로 열어둔 경우. (이때부터는 유틸리티와 경계가 모호해지지만, 특정 비즈니스 로직에 종속적이라면 여전히 헬퍼라고 부릅니다.)

### 4. 결론: 어떻게 구분해서 써야 할까?

이름을 지을 때나 코드를 짤 때 이렇게 자문해 보세요.

* **"이 코드를 옆 동네(다른 패키지, 다른 프로젝트)에서도 그대로 쓸 수 있나?"**
    * YES $\rightarrow$ **Utility** (별도 클래스로 분리, 예: `StringUtils`)
* **"이 코드는 지금 작성 중인 메인 로직이 너무 길어서 잘라낸 것인가?"**
    * YES $\rightarrow$ **Helper** (주로 `private` 메서드로 생성)
* **"이 코드는 외부에서 직접 호출하면 위험하거나 굳이 알 필요가 없는가?"**
    * YES $\rightarrow$ **Internal/Private**

이 개념들이 섞여서 쓰이는 것은 자연스러운 현상이니, 너무 칼같이 구분하려 스트레스받지 않으셔도 됩니다. 다만, **"범용성(Utility)이냐, 특정 문맥의 보조(Helper)냐"**만 구분해도 코드 설계가 훨씬 깔끔해집니다.
